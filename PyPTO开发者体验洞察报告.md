## 三、Top 5 痛点深度洞察

### 🥇 痛点 1:报错信息可读性差,定位链路断裂(影响整个调试旅程)

**信号强度**:描述中出现"报错" 252 次、"assertion failed" 35 次、"check failed" 4 次,直接的吐槽词包括"看不出原因""无法定位""报错信息不明确""不好定位"。

**用户原话(摘录)**:
- *"PyPTO 编译器 F21001 EINTERNAL 错误信息过于模糊,无法定位问题"*
- *"有些算子对 dtype、shape 等没有拦截,到调用底层指令的时候才报错,对用户来说报错信息很难读懂,而且也看不出具体是哪一行出错,kernel 实现比较复杂时很影响定位"*
- *"CHECK FAILED: t.GetDataType() == DT_INT32 报错看不出原因"*
- *"reshape 报错信息看不出来是哪里出错"*
- *"动态 shape tensor 在 loop 外 cast 操作报错日志不清晰"*
- *"where 的 condition 和 input 的 shape 不能广播时,报错信息不友好"*

**问题本质**:
1. **错误暴露点滞后**:类型/形状检查没在前端拦截,推迟到底层 IR 或编译期才抛出
2. **错误信息没有"源码映射"**:用户写的是 Python 高阶代码,但报错指向的是底层 Pass 名 + 内部断言
3. **错误信息缺乏"行动指引"**:只告诉你"哪里出错",不告诉你"怎么修"

**设计机会点**:
| 设计动作 | 期望收益 |
|---|---|
| **建立分层报错体系**:用户错误(USER_ERR)/ 编译器错误(COMPILER_ERR)/ 内部错误(INTERNAL_ERR),分类编号 + 不同呈现策略 | 让用户知道这是"我的问题"还是"工具的问题" |
| **错误信息三段式结构**:① What happened(自然语言)② Where(Python 源码行 + 内部 Pass 链路)③ Suggested Fix(可点击的修复建议) | 让"看不出原因"变成"看得懂、改得动" |
| **前置静态检查**:dtype/shape 等高频拦截点上移到 jit 装饰器解析阶段,失败即报 | 把"运行后崩"变成"提交前知道" |
| **错误码 → 文档反向链路**:每个 F21001 类编号自动挂接知识库链接 | 让一次报错带来一次学习 |

### 🥈 痛点 2:入门即受挫 — Hello World 与样例运行的"破窗效应"

**信号强度**:`hello_world` 出现 35+ 次,"样例报错""demo 报错""路径错"出现频次极高,其中超过 10 条 issue 直接指向"hello_world 跑不起来"。

**用户原话(摘录)**:
- *"运行 demo hello_world.py 出现问题"*
- *"用仿真模式跑 hello_world.py 用例报错"*
- *"https://gitcode.com/cann/pypto/blob/master/docs/invocation/examples_invocation.md 快速开始样例报错"*
- *"cd examples/hello_world :No such file or directory"*
- *"样例运行中 cd examples/hello_world 目录错误"*
- *"直接使用 pip 源上发布的 pypto 包,安装之后 example 用例执行报错,请整改"*
- *"运行 examples 文件夹下的 hello_world.py 文件出错,提示设置环境变量,需要创建一个 readme"*

**问题本质**:这是一个典型的"破窗效应" — 用户刚进门的第一个验证步骤就失败,会立刻引发对整个工具链的不信任。问题的根因有三层:
1. **文档与代码不同步**:目录结构变更,文档没跟上
2. **环境隐含依赖**:文档假设的环境变量、CANN 版本、C++ 编译器版本未显式说明
3. **缺少端到端可验证的"零依赖路径"**:没有"一键能跑"的方案

**设计机会点**:
| 设计动作 | 期望收益 |
|---|---|
| **"30 秒成功"快速开始**:提供预配好的 AscendHub 镜像 / Notebook,用户只需 `pypto.demo()` 即可看到成功输出 | 把"上手即受挫"翻转为"上手即正反馈" |
| **文档可执行化(Doctest-driven docs)**:所有文档代码示例进入 CI,文档与代码强一致 | 杜绝"文档说有的目录其实没有"这类问题 |
| **环境健康检查工具**:提供 `pypto doctor` 命令,自动诊断 CANN 版本、driver、g++、第三方库 | 把环境配置从"试错型"变成"诊断型" |
| **样例分层**:基础示例(零依赖)→ 进阶示例(单卡)→ 整网示例(多卡),分层入门 | 让不同阶段开发者都有合适的起点 |

### 🥉 痛点 3:文档"低错"与 API 说明缺失,理解成本极高

**信号强度**:56 条直接的文档反馈、61 条标题中含"文档"或 API 理解类问题。

**用户原话(摘录)**:
- *"文档存在低错"*
- *"当前 remainder 只支持 1-4 维,与文档描述不匹配"*
- *"【Green Light】TileShape 设置缺乏直观指导且约束复杂且难以记忆"*
- *"【Green Light】核心语法概念解释不足"*
- *"【Green Light】API 文档缺乏最佳实践"*
- *"【green light】pypto.loop_unroll 的 unroll_list 介绍难以理解,示例没有进行解释"*
- *"【green light】目前 api 文档没有一个汇总目录 readme"*
- *"vec_nbuffer_setting 参数的配置方式介绍不详细"*
- *"指向文件路径错误 / 文档跳转链接问题"*

**问题本质 — 文档反馈的三个层次问题**:
1. **基础层(低错)**:路径错、链接失效、与代码不同步 — 这是 CI 层面就能解决的
2. **结构层(无汇总)**:API 找不到入口、目录混乱 — 这是信息架构问题
3. **认知层(缺最佳实践)**:看得懂语法、不知道什么场景该用 — 这是教学法问题

**设计机会点**:
| 设计动作 | 期望收益 |
|---|---|
| **API 文档信息架构重构**:仿 PyTorch / JAX,每个 API 标配:签名 / 参数说明 / 形状约束 / 可运行示例 / 常见错误 / 相关 API | 解决"知道有但不会用"的问题 |
| **"Cookbook"机制**:为高频组合场景(Conv2d + TileShape、Matmul + 动态 shape)提供完整可运行的 recipe | 把"最佳实践"从口口相传变成可索引 |
| **文档质量门禁**:链接检查、代码片段执行、与 source 的描述一致性检查进 CI | 解决"低错"这一层的体验底线 |
| **TileShape 等关键约束的可视化助手**:在 IDE 插件中实时显示当前 tensor 的 tile 约束,而不是依赖文档记忆 | 让"难记的约束"变成"实时反馈" |

### 🏅 痛点 4:调试与精度定位 — 全旅程最痛环节

**信号强度**:精度问题相关 issue 64 条(标题)/ 118 条(描述);"调试""定位"类需求 125 条;`VerifyPass` 出现 16 次,精度工具相关 issue 大量出现。

**用户原话(摘录)**:
- *"开启精度工具后,发现 expandFunction pass 出现精度问题"*
- *"pypto.full 创建的 tensor 放在 loop 外有精度问题,放在 loop 里则精度正确,通过精度校验工具发现 TensorGraph 阶段有问题"*
- *"手动设置的 valid_shape 在 pass28 时被优化掉"*
- *"GDR 算子打开合轴优化之后,精度异常,当前精度工具尚未定位出来问题点"*
- *"【green light】Tiling 报错无法确定出错位置"*
- *"【green light】Error: ASSERTION FAILED: outDegree[opToIndex[op.get()]] == 0 报错什么意思?"*
- *"补充通信算子精度问题定位文档"*

**问题本质**:精度问题是 NPU 算子开发的核心难点,但目前开发者面对精度问题时:
1. **黑盒感强**:不知道是哪个 Pass 引入的精度损失
2. **二分定位成本高**:需要手动开关每个 Pass 来定位
3. **工具能力不全**:某些算子的精度工具尚未适配(如 index_add、TopK Radix Select)
4. **从工具到结论的"最后一公里"缺失**:工具告诉你"这里精度失败",但不告诉你"该怎么改"

**设计机会点**:
| 设计动作 | 期望收益 |
|---|---|
| **精度问题自动二分定位**:`pypto.jit(verify=True)` 默认开启全 Pass 校验,自动定位首个引入精度损失的 Pass | 把数天的人工 debug 变成数分钟的自动定位 |
| **Pass 级可视化(Pass Inspector)**:在 IDE 中以图形化方式展示每个 Pass 前后的 IR 变化与精度对比 | 把"黑盒优化"变成"白盒透视" |
| **精度问题模式库**:基于历史 issue 沉淀"valid_shape 被优化""loop 外 full 引发精度问题"等典型模式,在工具中主动识别并提示 | 将"踩坑"变成"前置告警" |
| **精度工具覆盖度治理**:建立每个算子/每个 Pass 的精度工具支持矩阵,补齐空白 | 让用户不再"工具不支持时只能盲改代码" |

### 🏅 痛点 5:性能调优靠"猜",缺乏从现象到决策的引导

**信号强度**:79 条性能直接相关 issue,泳道图相关咨询 20+ 条,"耗时长""太慢""性能劣化"等表述高频。

**用户原话(摘录)**:
- *"【green light】算子大规格静态场景编译太慢"*
- *"【green light】编译耗时长的原因如何分析?"*
- *"【greenlight】泳道图显示调度耗时过长"*
- *"[green light] 性能调试时,怎么通过泳道图查看是否达到性能瓶颈?"*
- *"算子端到端执行时间要 4s,泳道图上看只要 3us,请问是什么开销导致的,如何优化使用"*
- *"实现了一版 MHA,运行时长 20 分钟以上,请帮忙看下为什么会运行如此长的时间"*
- *"vec_nbuffer_setting 如何使用,是否存可以提供理论依据,或者 AI 辅助调优"*
- *"基于 AI 的 unroll_list 和 tile_shape 自动调优"*
- *"包含线性变换的算子出现 cube bound 现象,能否进一步优化?"*

**问题本质**:性能调优是一个高度依赖经验的环节,但目前开发者:
1. **看得到现象,猜不到原因**:泳道图能看到耗时,但不知道是 schedule、tile shape 还是数据搬运的问题
2. **不知道调优旋钮**:`vec_nbuffer_setting`、`stitch_function_inner_memory`、`unroll_list` 等参数复杂,缺乏决策依据
3. **缺反馈循环**:改了一个参数,要重新跑完才知道效果

**设计机会点**:
| 设计动作 | 期望收益 |
|---|---|
| **泳道图与源码双向跳转**:在泳道图中点击一个分块,跳转到对应的 pypto 代码行,反之亦然 | 解决"看到耗时,不知道改哪行" |
| **性能瓶颈自动诊断**:在泳道图上自动标注"cube bound""vector bound""通信 bound",并给出对应优化建议 | 把"靠猜"变成"被引导" |
| **AI 辅助调优(用户已自发提需求)**:基于历史调优数据,推荐 tile_shape、unroll_list 等参数初值 | 大幅降低调优学习曲线 |
| **What-If 性能预估**:在不实际运行的情况下,基于代价模型预估参数变更的性能影响 | 把"试一次跑几分钟"变成"实时预览" |

## 四、Green Light 专项体验观察

**Green Light 即 PyPTO 新前端**,在 4.5 个月内贡献了 122 条外部反馈(占 20%),其中:
- 缺陷 40 / 咨询 36 / 需求 29 / 文档 17
- **咨询占比 30%**(远高于其他模块的咨询占比),说明**新前端的概念心智模型尚未建立**

**用户在 Green Light 中反复问的问题**:
- 基础数据类型与构造:*"pypto 内怎么通过一个 list/tuple 数据构建新的 pyto tensor""pypto 有没有 tuple 的说法"*
- 新老前端迁移:*"pypto 新前端与老前端区别"*
- 接口语义:*"pypto.matmul 中是否存在 contiguous 的概念"*
- 工具链适配:*"IDE 中的 toolkit 如何进行功能、精度问题定位""toolkit 工具中的代码跳转会出现错位"*

**设计建议**:
1. **专门的迁移指南** — "From Old Frontend to Green Light",带前后对照代码
2. **概念地图(Conceptual Map)** — 解释 Green Light 的核心抽象(Tensor / Tile / Pass / Loop)及它们与 PyTorch/TensorFlow 中相似概念的异同
3. **新前端的 API 参考独立出来**,与老前端清晰隔离,避免心智污染

## 五、跨阶段贯穿性问题

除了单阶段痛点,有三类问题贯穿整个开发者旅程:

### 5.1 版本兼容性焦虑(描述中"版本/兼容"信号命中 277 次)
- CANN 版本、Driver 版本、PyPTO 版本、硬件型号(A2/A3/A5/910B/910C/950)的组合矩阵复杂
- 开发者经常在 issue 中花大量篇幅描述自己的环境配置,但仍可能"环境不对"
- **设计建议**:提供官方的"兼容性矩阵 + 自动检测工具",issue 模板预填环境信息

### 5.2 issue 模板形同虚设
- 当前 issue 模板有"Environment / 重现步骤 / 期望行为"等字段,但很多 issue 内容不完整
- **设计建议**:模板字段做成结构化输入(下拉框 / 命令插入),并引导用户先跑 `pypto doctor` 自动填充环境信息

### 5.3 优先级机制缺位(91% 无优先级)
- 开发者无法预期自己的反馈何时被处理
- 现有"自定义状态"有 13 种(已完成/进行中/已解决/待办的/已拒绝/...),内部含义不向外可见
- **设计建议**:对外暴露简化的状态流(已接收 → 排期中 → 修复中 → 已发版),让开发者知道自己处在哪一步

## 七、结论:从"能用"到"好用"的设计转向

PyPTO 当前的开发者体验现状,本质是**一个功能上能用、但每一个环节都让开发者付出额外认知负担的工具链**。614 条外部反馈呈现出一个清晰的画像:

> **开发者不是被某一个 bug 阻塞,而是被"信息缺口"持续消耗。**
>
> — 装的时候:不知道环境要求 → 缺口在"诊断"
> — 上手的时候:跑不通 hello_world → 缺口在"零依赖路径"
> — 写代码的时候:看不懂文档 → 缺口在"最佳实践"
> — 报错的时候:看不懂报错 → 缺口在"行动指引"
> — 调精度的时候:不知道哪个 Pass 出错 → 缺口在"自动定位"
> — 调性能的时候:不知道改什么 → 缺口在"瓶颈引导"

**设计的总方向**:从"提供工具"转向"提供路径"。

每一个体验设计动作的目标,都应该是**在开发者下一个动作之前,主动给出他需要的信息**——这是从"工具型 DX"到"引导型 DX"的核心跃迁。
